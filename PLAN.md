# План реализации ChatList

Поэтапный порядок работ на основании [PROJECT.md](PROJECT.md).

---

## Этап 1. Подготовка проекта

1. Обновить структуру каталогов и зависимости:
   - `main.py` — точка входа и GUI
   - `db.py` — работа с SQLite
   - `models.py` — бизнес-логика моделей и промтов
   - `network.py` — HTTP-запросы к API нейросетей
   - `.env.example` — шаблон переменных окружения для API-ключей
2. Добавить в `requirements.txt`: `httpx` (или `requests`), `python-dotenv`.
3. Зафиксировать схему БД в [DATABASE.md](DATABASE.md).

---

## Этап 2. Модуль базы данных (`db.py`)

1. Реализовать подключение к SQLite (файл `chatlist.db` в каталоге приложения).
2. Создать таблицы при первом запуске: `prompts`, `models`, `results`, `settings`.
3. Добавить CRUD-функции для каждой таблицы:
   - промты: создание, выборка, поиск по тексту и тегам;
   - модели: создание, обновление, получение активных (`is_active = 1`);
   - результаты: пакетное сохранение выбранных строк;
   - настройки: чтение и запись пар ключ–значение.
4. Инкапсулировать все SQL-запросы внутри `db.py` — остальные модули не обращаются к SQLite напрямую.

---

## Этап 3. Модуль моделей (`models.py`)

1. Описать dataclass/Pydantic-структуры: `Prompt`, `Model`, `Result`, `TempResult`.
2. Реализовать загрузку активных моделей из БД.
3. Реализовать работу с промтами: новый ввод, выбор из сохранённых.
4. Реализовать временную таблицу результатов в памяти (`list[TempResult]`):
   - поля: `model_name`, `response`, `selected`;
   - методы: добавление, очистка, получение выбранных, смена флага `selected`.
5. Реализовать сохранение выбранных результатов в БД через `db.py`.

---

## Этап 4. Модуль сети (`network.py`)

1. Загружать API-ключи из `.env` по имени переменной (`api_id` из таблицы `models`).
2. Реализовать базовую функцию отправки промта в одну модель (POST к `api_url`).
3. Добавить обработку ошибок: таймаут, HTTP-ошибки, отсутствие ключа в `.env`.
4. Реализовать параллельную или последовательную отправку промта во все активные модели.
5. *(Опционально)* Поддержка разных типов API: OpenAI-совместимый формат, DeepSeek, Groq.

---

## Этап 5. Основной интерфейс (`main.py`)

1. Заменить заглушку на полноценное окно PyQt6.
2. **Блок ввода промта:**
   - многострочное поле текста;
   - выпадающий список сохранённых промтов;
   - кнопка «Отправить».
3. **Таблица результатов** (временная, в памяти):
   - колонки: модель, ответ, чекбокс «Выбрать»;
   - обновление после получения ответов от всех моделей.
4. **Кнопка «Сохранить»** — запись строк с `selected = True` в таблицу `results`, очистка временной таблицы.
5. **Обработка нового запроса** — при вводе/отправке нового промта временная таблица очищается до получения новых ответов.
6. Индикатор загрузки на время ожидания ответов от API.

---

## Этап 6. Управление моделями и настройками (GUI)

1. Окно или вкладка «Модели»:
   - список моделей с полями: имя, URL, имя переменной ключа, активна/неактивна;
   - добавление, редактирование, включение/отключение модели.
2. Окно «Настройки»:
   - параметры из таблицы `settings` (таймаут запросов, путь к БД и т.д.).
3. Окно «История»:
   - просмотр сохранённых промтов и результатов из БД.

---

## Этап 7. Дополнительные функции (опционально)

1. Поиск и сортировка во всех таблицах GUI.
2. Экспорт выбранных результатов в Markdown и JSON.
3. Логирование запросов в файл (`logs/chatlist.log`).
4. Сборка исполняемого файла через PyInstaller (`build.ps1`).

---

## Этап 8. Тестирование и финализация

1. Проверить сценарий: ввод промта → отправка → выбор строк → сохранение.
2. Проверить сценарий: выбор сохранённого промта → повторная отправка.
3. Проверить поведение при ошибках API и отсутствии ключей в `.env`.
4. Обновить `README.md`: установка, настройка `.env`, запуск, сборка `.exe`.

## Этап 9. AI-ассистент для улучшения промтов

Реализация функции из [PROJECT.md](PROJECT.md), п. 7.

### Назначение

Помочь пользователю улучшить промт **до** массовой отправки в несколько моделей: одна выбранная модель (OpenRouter) анализирует текст и предлагает улучшенные варианты.

### Этап 9.1. Настройки и конфигурация

1. Добавить в таблицу `settings` ключи:
   - `assistant_model_id` — ID модели из таблицы `models` для ассистента (или имя модели OpenRouter, напр. `openai/gpt-4o-mini`);
   - `assistant_enabled` — включён ли ассистент (`1` / `0`);
   - *(опционально)* `assistant_system_prompt` — кастомный системный промт.
2. Расширить окно **«Настройки»** (`dialogs.py`):
   - выпадающий список модели-ассистента (только активные модели или все OpenRouter);
   - подсказка: «Для ассистента рекомендуется одна быстрая модель OpenRouter».
3. Значения по умолчанию при `init_db`: первая активная OpenRouter-модель или пусто (с предупреждением в UI).

### Этап 9.2. Модуль ассистента (`prompt_assistant.py`)

1. Создать dataclass `PromptSuggestion`:
   - `original` — исходный текст;
   - `improved` — улучшенная версия;
   - `alternatives: list[str]` — 2–3 варианта переформулировки;
   - `adaptations: dict[str, str]` — адаптации под тип задачи (`code`, `analysis`, `creative`).
2. Реализовать функцию `build_assistant_messages(user_prompt: str) -> list[dict]`:
   - системный промт с инструкцией вернуть **структурированный JSON**:
     ```json
     {
       "improved": "...",
       "alternatives": ["...", "...", "..."],
       "adaptations": {"code": "...", "analysis": "...", "creative": "..."}
     }
     ```
3. Реализовать `improve_prompt(prompt: str, model: Model, ...) -> PromptSuggestion`:
   - вызов существующего `network.send_prompt()` с системным + пользовательским сообщением;
   - парсинг JSON из ответа (с fallback при невалидном JSON — весь ответ как `improved`).
4. Обработка ошибок: пустой промт, нет модели-ассистента, ошибка API, таймаут.

### Этап 9.3. Интеграция с `network.py`

1. Добавить функцию `send_chat(model, messages, ...)` — отправка массива `messages` (system + user), если текущий `send_prompt` принимает только один user-текст.
2. Либо расширить `send_prompt` опциональным параметром `system_prompt: str | None`.
3. Логировать запросы ассистента через `app_log.py` (отдельный префикс `ASSISTANT`).

### Этап 9.4. Бизнес-логика (`models.py`)

1. Добавить методы в `ChatService`:
   - `get_assistant_model() -> Model | None`;
   - `improve_current_prompt(text: str) -> PromptSuggestion`.
2. Не сохранять результат улучшения в БД автоматически — только по действию пользователя («Подставить» / «Сохранить как промт»).

### Этап 9.5. Интерфейс (`main.py`, `dialogs.py`)

1. **Кнопка «Улучшить промт»** — рядом с полем ввода промта (под или справа от «Отправить»).
2. **Диалог `PromptAssistantDialog`**:
   - блок «Исходный промт» (read-only);
   - блок «Улучшенный промт» (read-only, выделяется);
   - список «Альтернативы» (2–3 варианта, радиокнопки или список);
   - вкладка или секция «Адаптации» (код / анализ / креатив);
   - кнопки:
     - **«Подставить в поле ввода»** — копирует выбранный вариант в `prompt_edit`;
     - **«Отмена»** — закрыть без изменений;
   - индикатор загрузки на время запроса к API.
3. **Фоновый поток** (`QThread` / worker) — не блокировать GUI при обращении к API (аналогично `FetchWorker`).
4. При пустом поле промта — предупреждение «Введите текст для улучшения».
5. Если модель-ассистент не настроена — сообщение с предложением открыть **Настройки**.

### Этап 9.6. Тестирование

1. Unit-тест: парсинг JSON-ответа ассистента (валидный / битый / пустой).
2. Unit-тест: `build_assistant_messages` формирует корректную структуру.
3. Mock-тест: `improve_prompt` при ошибке API возвращает понятное исключение.
4. Ручная проверка UI: улучшение → выбор варианта → подстановка в поле → отправка в модели.
